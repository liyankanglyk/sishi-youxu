# 四时有序 — 系统开发需求文档

---

## 项目概述

### 项目背景

**四时有序（Sishi Youxu）** 是一款基于 **艾森豪威尔矩阵**（Eisenhower Matrix）的个人任务管理工具。用户将任务按「重要性」和「紧急性」两个维度分配到四个象限，从而获得清晰的任务优先级视图。

Vue 3 + FastAPI 全栈应用，支持多用户、云端同步和管理后台。

### 目标用户

- 希望用艾森豪威尔矩阵管理个人任务的用户
- 需要跨设备同步的个人任务管理工具的用户
- 偏好简洁工具而非重型项目管理软件的用户

### 项目目标

- 提供直观、流畅的四象限拖拽体验
- **多用户支持**：账号注册、多渠道登录（账号密码/手机短信/邮箱验证码）
- **云端同步**：离线优先，云端实时同步，冲突可视化解决
- **管理后台**：用户管理、内容审核、审计日志
- 可作为 PWA 安装到设备主屏幕，支持离线使用

> 📌 移动端适配（Android / iOS）将在后续阶段独立开发。

## 功能需求

### 四象限画布

> 艾森豪威尔矩阵是本产品的核心 UI，所有任务以卡片形式分布在四个象限中。

#### 象限布局

| 象限 | 背景色 | 语义 | 用户行为 |
|------|--------|------|---------|
| Q1（右上）| `#F2A7B3`（粉）| 重要 + 紧急 | 立即处理 |
| Q2（左上）| `#B5D9C4`（薄荷绿）| 重要 + 不紧急 | 计划安排 |
| Q3（右下）| `#C5D5E8`（蓝）| 不重要 + 紧急 | 尽快完成 |
| Q4（左下）| `#FBE4A0`（黄）| 不重要 + 不紧急 | 有空再做 |

**坐标轴语义（整数型 v2）**：
- X轴（水平）：`urgencyLevel` 整数，-4 = 最不紧急，+4 = 最紧急（0 为分界线）
- Y轴（垂直）：`importanceLevel` 整数，-4 = 最不重要，+4 = 最重要（0 为分界线）
- 数值范围 `[-4, 4]`，后端建表为 `INT NOT NULL DEFAULT 0`，与 `init_db.sql` 严格一致

**象限判定（按整数正负）**：
- `urgencyLevel > 0` 且 `importanceLevel > 0` → Q1（重要 + 紧急）
- `importanceLevel > 0` 且 `urgencyLevel <= 0` → Q2（重要 + 不紧急）
- `urgencyLevel > 0` 且 `importanceLevel <= 0` → Q3（紧急 + 不重要）
- `urgencyLevel <= 0` 且 `importanceLevel <= 0` → Q4（不重要 + 不紧急）

**卡片定位**：画布归一化映射
- `left = (urgencyLevel + 4) / 8 * 100%`
- `bottom = (importanceLevel + 4) / 8 * 100%`
- 左下角锚定，等同 v1 的 `posX * 100%` / `posY * 100%`

| 编号 | 需求 | 优先级 |
|------|------|--------|
| CANVAS-001 | 四象限背景 + Q1-Q4 标签和含义说明 | P0 |
| CANVAS-002 | 十字分割线 + 箭头指示，坐标轴标注「重要/不重要」「紧急/不紧急」| P0 |
| CANVAS-003 | 任务卡片绝对定位，`urgencyLevel` 映射水平位置，`importanceLevel` 映射垂直位置（整数 -4..4）| P0 |
| CANVAS-004 | 拖拽移动：鼠标拖拽 + 触屏手势，实时更新位置 | P0 |
| CANVAS-005 | 象限自动判定：根据坐标自动归类到 Q1-Q4 | P0 |
| CANVAS-006 | 拖拽排序持久化：拖拽完成后，将 `sortOrder` 写入 Dexie 并同步到后端 | P0 |

| TASK-012 | 完成庆祝动画：canvas-confetti（尊重 reduced-motion）| P2 |

**重复任务生成规则（TASK-008 细则）**：
- **每日重复**：下一期 `dueDate = 当前 dueDate + 1 天`
- **每周重复**：下一期 `dueDate = 当前 dueDate + 7 天`
- **每月重复**：下一期 `dueDate = 当前 dueDate 的同一天 + 1 个月`（如 1月31日 → 2月28日/29日）
- 新任务 `createdAt`/`updatedAt` = 生成时的服务器时间
- 原任务 `completed = true`，不生成新任务而是克隆一条新记录

### 数据同步

**离线优先**：所有操作先写入本地 Dexie，立即响应 UI；500ms 去抖后异步推送到后端。

### 认证与账号

支持多登录方式绑定同一账号（多 Provider）：

| Provider | 说明 |
|----------|------|
| password | 账号 + 密码登录/注册 |
| phone_sms | 手机号 + 短信验证码（mock）|
| email_code | 邮箱 + 验证码（mock）|
| wechat | 微信授权登录（小程序/Mock）|
| wechat | 微信授权登录（小程序）|

**JWT 策略**：
- access_token：有效期 1 小时，短期有效
- refresh_token：有效期 7 天，入库支持撤销
- Logout：refresh_token 标记 revoked_at（DB），access_token 写入 Redis 黑名单（立即失效）
- refresh_token  revoked 后，客户端清除 token 并跳转登录页

**注册要求**：
- 必须有图形验证码（如滑动验证码）防爬
- 密码要求：最少 8 位，含大小写字母和数字
- 同一 IP 15 分钟内最多注册 10 次（通过 Redis 限流）

| 编号 | 需求 | 优先级 |
|------|------|--------|
| AUTH-001 | JWT access + refresh token | P0 |
| AUTH-002 | refresh token 入库支持撤销 | P0 |
| AUTH-003 | 多 Provider 登录（password/phone/email）| P0 |
| AUTH-004 | 账号绑定/解绑（同一账号绑定多个 Provider）| P0 |
| AUTH-005 | 图形验证码（获取验证码图片、校验验证码）| P0 |
| AUTH-006 | 微信登录（code2session + JWT）| P1（微信小程序扩展）|
| AUTH-007 | WebSocket ws-ticket 认证（替代 Sec-WebSocket-Protocol）| P1（微信小程序扩展）|

### 微信小程序（后续扩展）

> 📌 微信小程序作为独立项目开发，不与主项目（PWA/Capacitor）重叠。同一后端 API 通过不同前端适配支持多端。

**技术方案**：

| 组件 | 技术选型 |
|------|---------|
| 框架 | 微信小程序原生 + Taro（跨端扩展）|
| 状态管理 | MobX / Redux（适配小程序）|
| 本地存储 | 小程序 Storage（同步）+ IndexedDB（适配）|
| HTTP 请求 | wx.request（已兼容现有 API）|
| WebSocket | wx.connectSocket + URL 参数 ticket 认证 |
| 登录 | wx.login() → 后端 code2session → JWT |

**后端改造要点**：
- 新增 `wechat` AuthProvider
- 实现 `POST /api/v1/auth/wechat/login`（code2session + JWT 签发）
- 实现 `POST /api/v1/auth/ws-ticket`（WebSocket 认证）
- `sishiyouxu_auth_identity.provider_uid` 存储微信 openid

**与主项目共享**：
- 同一套后端 API（/api/v1/*）
- 同一套数据库
- 同一套 Redis（限流/缓存/JWT 黑名单）

## 非功能需求

### 性能

| 指标 | 要求 |
|------|------|
| 首屏加载 | 加载状态 < 2 秒 |
| 拖拽响应 | 拖拽帧率 ≥ 30fps，移动端不卡顿 |
| JS Bundle | gzip 后 < 500KB |
| 移动端优化 | 低端设备关闭 `backdrop-blur`（后续扩展）|

### 可用性

| 指标 | 要求 |
|------|------|
| 触屏支持 | 所有操作均有触屏等价交互 |
| 安全区域 | 全面屏设备 `safe-area-inset` 适配 |
| 状态持久化 | 侧边栏宽度、视图密度偏好存 localStorage |
| 加载状态 | 数据库初始化期间显示加载动画 |

### 安全

| 指标 | 要求 |
|------|------|
| JWT 安全 | access_token 短期有效，refresh_token 可撤销 |
| 数据隔离 | 用户只能访问自己的数据 |
| 输入校验 | 所有 API 输入使用 Pydantic 校验 |
| 防爬 | 注册需图形验证码，登录/注册接口 IP 限流（已实现） |
