## 调整说明（vs 原计划）

**原计划**按功能模块交替开发前后端（如 Phase 1 同时做管理端后端+前端），导致每个 Phase 都需要跨越前后端两个技术栈。

**新计划**采用**后端先行**策略：

1. **Phase 1 先完成所有管理端后端 API** — 集中精力把接口层、业务逻辑、数据访问层全部打通，前端可以用 Swagger/RapiDoc 即时验证
2. **Phase 1 接着完成管理端前端** — 后端 API 已全部就绪，前端只需对接已有接口
3. **Phase 2 完成用户端后端 API** — 任务/标签/通知/反馈/同步的全套业务逻辑
4. **Phase 3 完成认证与同步后端** — 多 Provider、微信 code2session、ws-ticket
5. **Phase 3+ 用户端前端交互** — 四象限拖拽、Dexie 同步引擎、WS 通知接收（最复杂的部分，留到最后打磨）

**优势**：
- 技术栈聚焦：每个 Phase 只涉及一种技术栈，减少上下文切换
- 接口稳定性：前端开发时后端 API 已全部稳定，无需等待后端跟进
- 联调效率：后端完成后，管理端和用户端前端都可以直接对接真实 API

## 开发优先级说明

**Phase 1 先做全部后端 API 的原因**：
1. 后端是系统的"骨架"，接口定了前端才能确定数据流
2. 所有接口集中开发，保持数据模型、错误码、响应格式的一致性
3. 通过 Swagger/RapiDoc 即时验证，不需要等前端页面
4. 前端开发时可以专注于交互和 UI，不被后端接口变更打断

**Phase 3 把用户端前端留到最后的原因**：
1. 用户端交互最复杂（拖拽画布/离线同步/手势操作），需要最多调试时间
2. 此时前后端接口已完全稳定，可以专注于体验打磨
3. 用户端是产品的"门面"，值得在最成熟的时机开发
