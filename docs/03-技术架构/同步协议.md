# 离线同步协议

---

## 设计目标

- **离线可用**：完全断网时仍可查看、新建、编辑、删除任务
- **在线同步**：恢复网络后自动推送本地变更、拉取远端变更
- **冲突解决**：基于 `updated_at` 的 last-write-wins（任务管理场景足够）
- **幂等**：每条 op 带 `opId`，重复推送安全

## 写入流程（乐观更新）

所有变更都先写入本地，再加入同步队列：

```ts
// 写本地 → 入队列 → 500ms 去抖 → 触发同步
await db.tasks.put({ ...task, localDirty: true })
await db.pendingOps.add({
  opId: crypto.randomUUID(),
  entity: 'task',
  action: 'upsert',
  payload: task,
  clientTs: Date.now(),
  status: 'pending',
})
syncEngine.schedule()  // 500ms 去抖
```

> ⚠️ `pendingOps` 存储在 IndexedDB 中，浏览器关闭即丢失（断电丢失）。`authSession` 同样仅存在于本地，不做服务端持久化，logout 时由服务端 API 处理 token 撤销。

---

## 同步触发时机

1. `online` 状态变化（断网 → 联网）
2. pendingOps 入队后去抖 500ms
3. 手动"立即同步"按钮
4. App 从后台切回前台
5. 启动时（如果有待同步操作）

## 推送变更：POST /api/v1/sync/push

### 请求

```json
{
  "ops": [
    {
      "opId": "client-uuid-1",
      "entity": "task",
      "action": "upsert",
      "payload": { ...完整 TaskRecord },
      "clientTs": 1737654321000
    },
    {
      "opId": "client-uuid-2",
      "entity": "tag",
      "action": "delete",
      "payload": { "uuid": "tag-uuid" },
      "clientTs": 1737654322000
    }
  ]
}
```

### 响应

```json
{
  "success": true,
  "data": {
    "results": [
      {
        "opId": "client-uuid-1",
        "status": "applied",
        "serverRecord": { ...更新后的 TaskRecord },
        "serverTime": "2026-06-25T10:30:00.123Z"
      },
      {
        "opId": "client-uuid-2",
        "status": "applied",
        "serverTime": "2026-06-25T10:30:00.124Z"
      }
    ],
    "serverTime": "2026-06-25T10:30:00.125Z"
  }
}
```

**status 取值**：
- `applied`：服务端已应用
- `conflict`：冲突，HTTP 返回 **200**（业务层面冲突，非 HTTP 错误），响应体含 `serverRecord`（驼峰）让前端选择；前端解决后重新推送
- `rejected`：格式错误或鉴权失败，HTTP 返回 400/401

> 字段命名与 `API接口文档.md §同步接口` 完全对齐（`serverRecord` / `serverTime` / `opId`），客户端只需维护一份 TypeScript 类型即可。

## 冲突判定

```python
def apply_upsert(existing, payload):
    if existing is None:
        return repo.create(payload)

    if payload.updated_at > existing.updated_at:        # DB 字段名
        # 客户端较新：覆盖
        return repo.update(payload)
    elif payload.updated_at < existing.updated_at:
        # 客户端较旧：冲突，返回服务端版本
        # ConflictResult.serverRecord 对应 API 响应字段 serverRecord（camelCase）
        return ConflictResult(serverRecord=existing)
    else:
        # 时间戳相同：幂等，直接返回
        return AppliedResult(serverRecord=existing)
```

### sortOrder 多设备冲突

当同一象限内的多个任务 `sortOrder` 发生冲突时（两设备同时修改导致 sortOrder 重复或交叉），按以下规则处理：

1. **同象限内 sortOrder 冲突**：以 `updated_at` 较新的为准，保留其 sortOrder；被覆盖的客户端需重新计算受影响任务的 sortOrder（向后顺延）
2. **跨象限移动后 sortOrder**：移动操作视为一次独立的 `upsert`，以 `updated_at` 判决；目标象限内其他任务的 sortOrder 不受影响
3. **批量操作中的 sortOrder**：批量移动/删除的 sortOrder 变更通过独立 op 推送，统一经过上述冲突判定

## 拉取变更：GET /api/v1/sync/pull

```
GET /api/v1/sync/pull?since=2026-06-24T00:00:00.000Z&entities=task,tag,taskTag,checklistItems
```

- `since`：上次拉取时间，返回该时间之后变更的记录
- `entities`：逗号分隔，支持 `task`、`tag`、`taskTag`、`checklistItems`；不传则返回全部 4 种实体

**响应**：

```json
{
  "success": true,
  "data": {
    "tasks": [...],
    "tags": [...],
    "taskTags": [...],
    "checklistItems": [...],
    "serverTime": "2026-06-25T10:30:00.000Z"
  }
}
```

软删除的记录也会返回（含 `deletedAt`），前端从本地移除。

## 状态机

```
pendingOps.status 状态转移：

pending ──(syncEngine 处理)──→ syncing
                                  │
                                  ├──(全部 applied)──→ synced（30s 后清理）
                                  │
                                  ├──(部分 conflict)──→ conflict
                                  │                       │
                                  │                       └─(用户选择)──→ pending
                                  │
                                  └──(错误)──→ failed
                                          │
                                          └─(重试 5 次)──→ abandoned
```

---

## 错误处理

| 错误类型 | 处理 |
|---|---|
| 网络断开 | ops 保持 `pending`，下次联网自动重试 |
| 401 | 调 `/auth/refresh`，刷新成功后重试 |
| 5xx 服务端错误 | 指数退避重试（1s → 2s → 4s → 8s → 16s），5 次后放弃 |
| conflict | 弹窗让用户选择"保留本地"或"使用服务端" |

---

## 性能限制

- push 批量：单次最多 100 条 ops
- pull：单次最多 500 条记录
- 触发去抖：500ms
